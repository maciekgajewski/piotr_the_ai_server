One conversation, one bridge coroutine

The long-running coroutine is the conversation's state holder. It shuffles typed messages between exactly one input conversation and exactly one agent conversation. Multiple inputs simply run multiple independent bridges.

The fleet view: concurrency by replication

A websocket connection and a microphone are persistent input sessions. Each may have zero or one active conversation. When both are active, they do not share a conversation state machine: each gets its own bridge coroutine and isolated agent conversation.

input-owned bridge-owned agent-owned

One bridge: a narrow, enforceable boundary

During startup the bridge receives sealed input and agent factories plus typed context; during the active exchange it sees only the two scoped conversation interfaces. It never knows whether the input is a websocket, microphone, or future medium, or which concrete agent is behind the agent interface.

Illegal directions are not merely documented—they are absent from the interfaces. Local framing is checked by each conversation object; cross-side ordering is checked once, in the bridge.

A complete turn, including streamed speech

Inputs enter the core only as complete user messages. Assistant text moves in the other direction as a stream, so a future microphone renderer can begin speaking without waiting for the full reply.

If the user responds

UserMessage crosses the same bridge and begins the next agent turn.

If the offer expires

The input sends FollowUpTimedOut; the bridge ends the conversation with a typed reason.

Policy stays with the component that has the facts

The core describes meaning and ordering. Bindings decide how that meaning is transported or rendered. This keeps microphone timing out of websocket code and websocket framing out of agents.

Cancellation is universal; failures are isolated by trust boundary

An input may cancel from every non-terminal state. The bridge promptly cancels agent work, stops new output, aborts an open stream, and closes both scoped conversations. What happens next depends on who failed—not on a free-text error message.

A protocol violation by an external websocket client is session-fatal: reject it with a typed response where safe, then close. Invalid output from an agent is conversation-local. A broken bridge invariant is an internal defect and terminates the input session after best-effort cleanup. If several results are already ready together, the bridge uses the ratified terminal-priority order rather than scheduler luck.

Bridge-owned startup; input-owned backpressure

The bridge opens both scoped sides

request = await input_session.accept_conversation()

await bridge_conversation(
    request,
    input_session,
    agent,
    context_provider,
)

# Inside the bridge, already in STARTING:
conversation_id = new_conversation_id()
context = await resolve_context(...)

async with input_session.open_conversation(
        request, context
    ) as input_conversation:
    async with agent.open_conversation(
            context
        ) as agent_conversation:
        await run_state_machine(...)

The bridge exists before context resolution or either __aenter__, so it owns startup cancellation, failure classification, and exact-once cleanup.

A slow input pushes back by blocking

1

The agent offers one ordered assistant event.

2

The bridge validates it and awaits the input sink's send.

3

A slow device may use its own bounded buffer. When full, send simply blocks.

4

The blocked bridge stops accepting agent output, so pushback reaches the producer without a bridge queue.

The input sink serializes complete() and priority abort(). Exactly one commits. Abort discards only input-buffered, unrendered output; anything already presented remains visible or spoken.

What changes—and what deliberately does not

Changed together

  • Core messages, interfaces, and state machine
  • Every Agent implementation
  • Websocket JSON server and repository clients
  • Microphone manager-to-conversation binding
  • Normative documentation and conformance tests

Preserved in T-004

  • Normative microphone manager-to-driver protocol
  • Concrete drivers and firmware behavior
  • T-002 audio-progress correlation fix
  • T-003 pre-roll and stop-race behavior
  • Websocket background reading for heartbeat and liveness

Future microphone work may add true streaming TTS or stronger playback-abort acknowledgement. T-004 records those possibilities but does not silently expand the driver protocol.
