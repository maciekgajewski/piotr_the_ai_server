One conversation, one bridge coroutine

The long-running coroutine is the conversation's state holder. It shuffles typed messages between exactly one input conversation and exactly one agent conversation. Multiple inputs simply run multiple independent bridges.

The fleet view: concurrency by replication

A websocket connection and a microphone are persistent input sessions. Each may have zero or one active conversation. When both are active, they do not share a conversation state machine: each gets its own bridge coroutine and isolated agent conversation.

input-owned bridge-owned agent-owned

One bridge: a narrow, enforceable boundary

The input adapter first returns a completely ready InputConversation. The bridge then adopts its ID, starts watching input control, resolves typed agent context synchronously, and races that watcher against AgentConversation entry and every user-message handoff. It never knows whether the input is a websocket, microphone, or future medium, or which concrete agent is behind the agent interface.

Illegal directions are not merely documented—they are absent from the interfaces. Local framing is checked by each conversation object; cross-side ordering is checked once, in the bridge.

A complete turn, including streamed speech

Inputs enter the core only as complete user messages. Assistant text moves in the other direction as a stream, so a future microphone renderer can begin speaking without waiting for the full reply.

If the user responds

UserMessage crosses the same bridge and begins the next agent turn.

If the offer expires

The input sends FollowUpTimedOut; the bridge ends the conversation with a typed reason.

On a microphone, speech start is the local commit point. Speech detected at or before timer expiry cancels and suppresses that timeout while STT finishes. If expiry commits first, the adapter emits one FollowUpTimedOut and suppresses later speech. For websocket input, transport handoff opens an adapter-owned one-value gate. A fast validated reply waits there until the bridge has entered and acknowledged WAITING_FOR_FOLLOW_UP; it never crosses early or queues in the bridge.

Policy stays with the component that has the facts

The core describes meaning and ordering. Bindings decide how that meaning is transported or rendered. This keeps microphone timing out of websocket code and websocket framing out of agents.

Cancellation is universal; failures are isolated by trust boundary

An input may cancel from every non-terminal state. The bridge promptly cancels agent work, stops new output, aborts an open stream, and closes both scoped conversations. What happens next depends on who failed—not on a free-text error message.

A protocol violation by an external websocket client is session-fatal: reject it with a typed response where safe, then close. Invalid output from an agent is conversation-local. A broken bridge invariant is an internal defect and terminates the complete server process non-zero after bounded best-effort notification and logging. If several results are already ready together, the bridge uses the ratified terminal-priority order rather than scheduler luck.
Ordinary application shutdown stays outside the event vocabulary. The application stops admission and closes every InputSession; bridges observe ordinary InputSessionClosed, cancel Agent work, and clean up within one global deadline. Deadline expiry or a second signal hard-exits non-zero.

Expected context outcome

CONTEXT_REJECTED carries a required closed rejection code. It and CONTEXT_UNAVAILABLE end only this conversation, so the input session may become ready again.

Broken provider contract

Malformed results or unexpected exceptions are INTERNAL_FAILURE: bounded terminal delivery is attempted, then the complete server process terminates non-zero.

Ready input first; input-owned backpressure

The input returns a usable conversation

async with input_session.accept_conversation(
    ) as input_conversation:
    await bridge_conversation(
        input_conversation,
        agent,
        context_provider,
    )

# Inside the bridge, now in STARTING:
conversation_id = (
    input_conversation.context.conversation_id
)
input_control = create_task(
    input_conversation.receive_control()
)
context_result = resolve_agent_context(...)
startup = select_context_or_input_control(
    context_result,
    input_control,
)
if startup.ends:
    await finish_startup(startup)
    return

entry = await race_agent_entry(
    agent.open_conversation(startup.context),
    input_control,
)
if entry.ends:
    await finish_startup(entry)
    return
delivery = await race_user_delivery(
    entry.agent_conversation,
    input_conversation.initial_message,
    input_control,
)
if delivery.ends:
    await finish_delivery(delivery)
    return
await run_state_machine(
    entry.agent_conversation,
    pending_input_control=input_control,
)

Before the input context returns, setup failures are input-session-local and no core conversation exists. After it returns, the bridge observes input control continuously—including while either initial or follow-up input waits for Agent acceptance. Input control wins simultaneous readiness; every losing task is joined.

A slow input pushes back by blocking

1

The agent offers one event through a zero-capacity rendezvous.

2

The bridge validates it and awaits the input sink's send.

3

A slow device may use its own bounded buffer. When full, send simply blocks.

4

The blocked bridge stops receiving; the rendezvous blocks agent production without either side owning an output queue.

Every input binding declares its commit point. abort() wins only before completion commits. A websocket frame commits at transport handoff—even if flow-control drain is still blocked—while a microphone may discard anything still uncommitted in its own bounded renderer buffer. An in-flight websocket writer is serialized and allowed to finish or close the session; it is never reinterpreted as unsent. Agent cancellation is acknowledged only after all model, tool, producer, and background work has stopped; missing the entry or cancellation deadline terminates the complete server process non-zero.

Bounded connections; bounded graceful shutdown

Websocket capacity is reserved before upgrade, and follow-up waiting has a server resource lease which is separate from the client's semantic timeout. Application shutdown uses the same sealed InputSession close path and one global deadline.

Explicit resource configuration

max_connections, capacity Retry-After, follow-up idle lease, and shutdown grace period are required positive configuration values. There are no hidden defaults.

Hard bound remains real

Lease expiry closes with 1013 Try Again Later and never fabricates FollowUpTimedOut. A second shutdown signal or global deadline expiry forces immediate non-zero process exit.

The websocket adapter—not the bridge—holds at most one validated reply that arrives after handoff while flow-control drain is blocked. The matching commit-token acknowledgement opens the gate only after the bridge state is ready. Cancellation, input failure, and connection close bypass the gate and retain their normal priority; every terminal path clears the held reply.

What changes—and what deliberately does not

Changed together

  • Core messages, interfaces, and state machine
  • Every Agent implementation
  • Websocket JSON server and repository clients
  • Microphone manager-to-conversation binding
  • Server signal handling, explicit resource limits, and graceful cleanup
  • Normative documentation and conformance tests

Preserved in T-004

  • Normative microphone manager-to-driver protocol
  • Concrete drivers and firmware behavior
  • T-002 audio-progress correlation fix
  • T-003 pre-roll and stop-race behavior
  • Websocket background reading for heartbeat and liveness

Future microphone work may add true streaming TTS or stronger playback-abort acknowledgement. T-004 records those possibilities but does not silently expand the driver protocol.
